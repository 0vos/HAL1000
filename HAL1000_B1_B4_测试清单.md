## 〇、部署 & 运行方式说明（先看这里）

### 1. 怎么把代码传上服务器

把整个 `agent/` 文件夹（本地路径 `/home/node/.openclaw/workspace/code/agent`）传到服务器 `/root/siton-tmp/HAL1000/agent/`，覆盖原有内容即可。**注意排除以下几个目录，没必要传（体积大或是本地生成的临时数据）**：

传完后在服务器上装依赖（`requirements.txt` 刚补了 `pillow`，需要重新装一次）：
```bash
cd /root/siton-tmp/HAL1000/agent
pip install -r requirements.txt
```

### 2. `bash run_hal.sh` 是什么，什么时候用

`code/run_hal.sh` 就是测试清单里 **B1 表 #9 那条交互命令的快捷方式**，内容等价于：
```bash
python3 hal_chat.py --mode prompt_json --model_path /root/siton-tmp/HAL1000/Qwen3.5-4B --toolset all_tools --verbose
```
**用于测试表格"二、核心问题"里所有需要交互输入的项**（写代码、追问、/branch、/undo 等）。跑起来之后就是一个可以持续对话的终端，不需要每次重新跑命令，在同一个进程里连续输入不同的 prompt 测不同能力即可。

### 3. 只是"发个 prompt 看回答"的测试，用什么命令

分两种情况：

**情况 A：想在同一个会话里连续问多个问题（推荐）**
```bash
cd /root/siton-tmp/HAL1000/agent/code
bash run_hal.sh
# run_hal.sh 内部已经写死了 --model_path，不需要额外 export HAL_MODEL_PATH
# 等价于：python3 hal_chat.py --mode prompt_json --model_path /root/siton-tmp/HAL1000/Qwen3.5-4B --toolset all_tools --verbose
```
进去之后就是个聊天界面，跟"二、核心问题"表格里逐条输入即可，退出用 `/quit` 或 Ctrl+C。

**情况 B：单条 prompt，跑一次就要结果，不需要交互（比如 B1/B4 测试表里那些 `b1_agent_runtime.py` / `b4_local_agent_llm.py` 命令）**
直接照抄测试表里给的完整命令，这些命令本身就是"输入一个 JSON 文件 → 跑一次 → 结果写到 outdir"，不需要 `run_hal.sh`，也不需要交互。

**两者的区别**：`hal_chat.py`（`run_hal.sh`）是完整 REPL 交互，有 DAG/裁判/记忆/人在环确认这些框架能力；`b1_agent_runtime.py`/`b4_local_agent_llm.py` 是单次调用的独立脚本，用来验证 B1/B4 模块本身的基础功能是否正确（更贴近 PPT 里"基础要求"的验收方式）。**这次 HAL1000 框架升级的能力（DAG/裁判/NUDGING/记忆/人在环等）几乎都只能通过 `hal_chat.py` 交互测出来**，所以"二、核心问题"那张表是最重要的部分。

## 一、结论先行

**B1-B5 的基础要求 + 进阶要求，代码文件层面全部存在实现**。
在 PPT 规定的进阶要求之上，HAL1000 框架（`hal_chat.py`）又做了额外的能力升级：

| PPT 未要求，HAL1000 额外做的 | 说明 |
|---|---|
| DAG 并发调度 | LLM Planner 拆解任务图，无依赖节点并发执行 |
| LLM 语义裁判 + NUDGING | 驳回后自动重推，两条路径（DAG/FSM）都接入 |
| Episodic Memory 三层记忆 | SQLite + MiniLM/BM25 检索 |
| Artifact Registry | 产物版本管理 + 真正回滚 |
| Session 持久化 + 分支/撤销 | `/branch` `/undo N` |
| 人在环确认 | DAG 规划完成后展示计划，等待人工确认/修改/取消 |

**关于 mock vs 真实模式**：`mock` 是纯规则模拟，只用来验证代码框架能跑通（不调用模型），**不能代表真实规划/生成能力**。下面所有命令统一改为 `prompt_json`（真实模型）模式。少数命令保持 `fixture`/`mock` 是有意为之（见备注），因为它们测的是机制本身（checkpoint/batch runner 的状态管理），不是 LLM 能力。

## 二、核心问题：我问 Agent 什么，能测出什么？

这是最重要的部分。**不同的 prompt 会触发不同的代码路径**，看输出日志里出现了什么关键字，就能判断对应能力是否生效。

### 2.1 终端交互测试（`hal_chat.py`）—— 推荐从这里开始

```bash
cd /root/siton-tmp/HAL1000/agent/code
export HAL_MODEL_PATH=/root/siton-tmp/HAL1000/Qwen3.5-4B
python hal_chat.py --mode prompt_json --toolset all_tools --verbose
```

进入交互界面后，按下表依次输入，每次观察日志关键字：

| 你问什么 | 测的是什么能力 | 该看到的日志关键字 | 判断标准 |
|---|---|---|---|
| `写一个快速排序算法，保存到 sort.py` ✅| LLM Planner 任务拆解 + DAG 并发 + 人在环确认 | `[规划器(LLM)]`、`任务计划确认` 弹窗、`[▶ DAG]` | 看到计划展示（思路+步骤），回车确认后看到多个节点并发执行 |
| （对上面的计划确认框）输入 `改成用归并排序，并且加上时间复杂度注释` ❌| 反馈重新规划 | `[规划器] 根据反馈重新规划（第 1 次）...` | 第二次展示的计划步骤应体现出"归并排序"字样 |
| （对确认框）输入 `n` ✅| 取消机制 | `[系统] 已取消本次任务` | 不应有任何工具被执行 |
| `帮我写一个二分查找，然后再写一个字符串反转函数，都保存到 utils/ 目录下` ✅| 多管道 DAG（识别成 2 组独立任务） | 计划里应出现 6 个左右节点（两组各 3 步），部分并发 | 步骤列表里能看到 2 个不同的 save_path |
| `解释一下什么是动态规划` ❓| Planner 判断为纯问答，fallback 到 FSM（不该生成 DAG） | 不应出现"任务计划确认"，应直接走 FSM 循环给出回答 | 若仍弹出确认框说明 Planner 误判，需反馈 |
| `帮我读取 data/algorithms/bubble_sort.py 的内容并总结一下逻辑` ❌| file_reader 工具 + FSM 裁判 | `[裁判]` 日志、工具调用 `file_reader` | 回答里应引用文件真实内容，不是编造的 |
| `刚才写的快速排序，能不能加个去重功能？`❌ (紧接上文追问) | Episodic Memory / working memory 记忆连续性 | 无特定日志，看回答是否"记得"上文内容 | 若完全不认识"刚才"指的是什么，说明记忆没接上 |
| （连续问 8-10 轮不同问题后）随便问一句✅ | Context 压缩触发 | `[系统] 对话历史过长，已自动压缩` 或 `[记忆] working memory 已修剪` | 出现这行说明压缩生效 |
| `/branch` ✅| Session 分支 | 提示已创建分支 session | `sessions/session_xxx.json` 新文件生成 |
| `/undo 1` ✅| 撤销机制 | `[撤销]` 相关提示 | 消息数量应减少 |
| `/quit`✅ 后重新用 `--resume <session_id>` 启动❌ | Session 恢复 | `[系统] 已恢复会话 xxx（N 条消息）` | 历史消息应该还在 |

### 2.2 如何验证"节点级 Retry"和"DAG→FSM fallback"（较难主动触发，给你两个可靠方法）❓

```bash
# 方法1：故意让 DAG 规划一个会失败的读文件任务
python hal_chat.py --mode prompt_json --toolset all_tools --verbose
# 输入：帮我读取 /root/不存在的文件.txt 并总结内容
```
预期：`file_reader` 节点会失败，若所有节点都失败会看到 `[DAG→FSM] 所有节点均失败...切换到 FSM ReAct 路径`。

### 2.3 如何验证"多阶段模型切换"

同一个 `hal_chat.py` 交互里，输入一个能触发 DAG 的请求，`--verbose` 模式下留意：
```
[model_switch] stage=plan → ...❌
[model_switch] stage=execute → ...✅
[model_switch] stage=summarize → ...✅
```
三个 stage 都应该打印（虽然 `model_roster.yaml` 里三者目前指向同一个 `model.yaml`，只是生成参数不同：plan 用少 token 快速生成，summarize 用 temperature=0 稳定输出）。

## 五、为什么有些命令我故意保留 mock/fixture，不是漏改

**这三类命令本来就不该用真实模型跑**（不是我改漏了）：

1. **`b1_agent_runtime.py --input runtime_input_checkpoint.json`**：这份输入文件本身标注 `execution_mode: fixture`，用预设的 `ai_messages`/`tool_messages` 模拟一轮对话，目的是隔离测试"checkpoint 保存/恢复机制"，如果换成真实模型，checkpoint 的中断点就不可控（无法保证每次都在同一步中断），测试就失去意义。

2. **`b1_batch_runner.py --batch_input batch_input.json`**：文件里每条任务都写死 `execution_mode: fixture`，测的是"批量调度器能不能正确遍历多条任务并汇总结果"，跟模型好坏无关。

3. **`b4_multi_tool.py` / `b4_plan_execute.py`**：这两个脚本的文档注释明确写着 `mock mode, no real LLM required`，是 PPT 进阶要求里"演示某个数据结构/流程"的独立小样例，本来就没有对接真实模型的代码路径。**如果你想测真实模型的多工具调用/Plan-and-Execute 能力，应该用 `hal_chat.py` 的 DAG 路径**（我在 B1 表 #9 已经给了对应的真实模型测试方法）。

如果你希望这两个脚本也支持真实模型，我可以加一个 `--mode prompt_json` 分支，但这是额外开发工作，不是"改命令"就能解决的。需要的话告诉我。

## 六、视觉/读图功能（代码已写完，未在真实环境验证——见下方说明）

**Qwen3.5-4B 本身就是原生多模态模型**（architecture: `Qwen3_5ForConditionalGeneration`），不需要额外训练视觉编码器。已实现以下文件：

- **`skills/image_reader.py`** — 图片文件校验 + 元信息提取
- **`skills/image_qa.py` + `b4_local_agent_llm.py::generate_vision_answer`** — 真实视觉问答
- **`code/vision_smoke_test.py`**（新增）— 独立最小验证脚本，分 3 步排查问题出在哪一层

**⚠️ 重要澄清**：之前说"本地验证通过"，实际上本地环境没有 torch/transformers，代码在 `import torch` 那一行就 ImportError 退出了——**AutoProcessor 加载、模型加载、图文问答推理这些真正核心的逻辑，一行都没有跑过**。这份代码是照官方 API 写的推断实现，不是验证过的实现。诚实地说：唯一验证过的是"mock 模式返回占位字符串"和"参数校验逻辑"，这些跟视觉推理本身没关系。

真正的风险点：
1. `transformers==5.12.1`（当前 `requirements.txt` 锁定版本）是否认识 `Qwen3_5ForConditionalGeneration` 这个新架构，如果这个版本发布时间早于 Qwen3.5 支持合入，会直接报错
2. `processor.apply_chat_template` 对图文混排 content 格式的渲染是否符合预期
3. `processor(text=, images=)` 的返回是否能直接喂给 `model.generate()`

**`requirements.txt` 顺带补了一个缺失依赖**：之前没有 `pillow`，已加上 `pillow>=10.0.0`。
